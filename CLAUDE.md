# CLAUDE.md: the SLAPTURE capture system

##  *Understanding the premise of this system*

*This system is all about intelligent routing and not bothering the user. Consider this in depth in order to make smart design decisions. The core process goes as follows:*
- *input comes in*
  - *currently assumed to be directly from human, but could later be auto-piped from some other event perhaps*
- *maybe it gets immediately automatched & routed*
- *if not, LLM systems immediately route it somewhere sensible (could be many different integrations/endpoints)*
- *the LLM systems manage the automatching*
- *ultimately we aim to have the integrations itself be generated by a claude code instance that gets invoked in the right circumstances!*
- *the whole system is self-correcting and self-documenting with high-context and backwards-compatibility-sensitivity, and with human user oversight*

*Based on that logic, it should be OBVIOUS that in the following 'wait for pattern' example the correct answer is "Soft route".*
*Option 1 fails to reliably deliver things where they should go, as best it can.*
*Option 3 bothers the user.*

```
On the "wait for pattern" behavior - how should interim inputs be handled?

Option 1: Inbox + clustering
Description: Goes to inbox, system clusters similar items, proposes route after N similar
────────────────────────────────────────
Option 2: Soft route
Description: Creates a "draft" route that doesn't auto-execute - needs human approval or N confirmations
────────────────────────────────────────
Option 3: Ask immediately
Description: First novel input triggers "I see you want to log gwen memories - is this a pattern you'll repeat?"
```

## Dev tips
- use `pnpm`, not `npm`
- `pnpm dev` for development (which runs `tsx watch` under the hood)
- do test-driven development
- always run tests before concluding things are good, not just the build step!
  - run E2E tests too, not just unit tests
- **NEVER claim a bug is fixed without verifying through Playwright E2E against the running server.**
  - Unit tests and direct function calls are NOT sufficient — they bypass the HTTP layer, auth, Firestore, and the widget UI. If a user reports a bug through the widget, your verification MUST go through the widget (Playwright). A passing unit test does not mean the bug is fixed.
  - imagine that you're a human developer, and you get assigned to fix a bug described in terms of the end-user experience. you need to go and test the actual end-user flow just like the person who assigned it to you did/will! and playwright is your tool for this
- if you're executing a plan and you've thoroughly tested it, go ahead and commit
- use your superpowers
  - prefer subagent-driven execution
- your tools, for security reasons, seemingly won't show the existence of a .env file. don't worry about it. it's there.
- do your best to avoid running multiple commands at once (eg pnpm)
  - eg DON'T RUN THIS: pkill -f "node dist/index.js" 2>/dev/null; sleep 1; ANTHROPIC_API_KEY="$ANTHROPIC_API_KEY" pnpm start & timeout: 30s sleep 3
  - and DON'T RUN THIS: cat > ./tmp/test-prompt.ts << 'EOF'  [file contents]
- relatedly, use actual temp scripts, since they work better for permissions
  - eg DON'T RUN THIS: pnpm tsx -e "..."
    - instead, make the temporary script in `./tmp` and then run it. if you need to make a specific
- if you need to write to tmp, write to `./tmp` so you don't need to ask for permission to write to `/tmp`

## Deployment

Deployed on Google Cloud Run (`slapture` service, `us-east1`).

```sh
gcloud run deploy slapture --source . --region us-east1
```

**IMPORTANT:** When updating env vars, ALWAYS use `--update-env-vars` (merges), NEVER `--set-env-vars` (replaces all and wipes existing ones).
